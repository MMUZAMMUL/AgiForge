## 2. Security

**HIGH — XSS via `innerHTML` + incomplete markdown escaping.** The `md()`
helper (around line 1151–1165) escapes `<`/`>`/`&` inside inline code and
`<pre>` blocks, but **not** raw HTML tags appearing in ordinary paragraph
text. LLM output is then inserted via `innerHTML=md(output)` at multiple
call sites: pipeline execution (~1605, 1612, 1622), chat rendering
(~1911–1923), debate (~2446, 2454), benchmark (~2540, 2557). Since agent
responses are LLM-generated and not controlled by this app, a response
containing something like `<img src=x onerror="...">` outside a code
fence would execute. There is no CSP meta tag to backstop this. **Fix in
Phase 0**: escape all literal `<`/`>` in markdown source before applying
the limited markdown→HTML transform, or special-case only the constructs
`md()` actually needs to turn into tags (code fences, bold/italic, links)
rather than passing arbitrary text through.

**MEDIUM — no CSP.** No `Content-Security-Policy` meta tag exists. Given
the app is 100% inline script/CSS by design, a meaningful CSP needs
`'unsafe-inline'` carve-outs today; this gets easier once Phase 0 moves
CSS/JS to external files, at which point a real CSP (no `unsafe-inline`)
becomes possible and is recommended as a Phase 0 follow-up.

**MEDIUM — provider API keys in plain `localStorage`.** Expected for a
no-backend client app (keys are user-supplied, not generated by this
project), but it raises the stakes of the XSS finding above: if the XSS
gap is exploited, stored Groq/Cerebras/Gemini/OpenRouter keys and the
Brave Search key are readable by the injected script. Closing the XSS gap
is the actual mitigation; there is no practical way to protect secrets in
`localStorage` from same-origin script.

**LOW — stale Supabase reference in `sw.js`.** Line 29 lists
`supabase.co` in the service worker's "never cache, always hit network"
host list. Harmless today (nothing calls Supabase), but it's a leftover
from the removed auth system — see §3 and the Documentation Report for
the fuller stale-Supabase picture.

**Clean**: no `eval`/`new Function`, no hardcoded API keys/secrets in
source, no Supabase code left in `index.html` itself (the removal of
auth/Supabase/Pro code was otherwise clean).

## 3. Dead code

Code-level dead code is minimal — the codebase is actively maintained and
the Supabase/auth removal was clean at the `index.html` level. The real
dead-code problem is one level up, in docs and schema files that
describe functionality the app no longer has:

- `supabase/schema.sql` (75 lines: `profiles`, `reviews`, `app_config`
  tables with RLS) — fully intact schema for a feature that was removed
  from the app. Nothing in `index.html` references it.
- `MONETIZATION.md` and `CLAUDE.md` describe Supabase-backed
  auth/Pro-plan/reviews as active features (they are not — see
  `04-DOCUMENTATION-REPORT.md` for exact lines).
- `sw.js:29` `supabase.co` entry (see §2).

**Recommendation**: either delete `supabase/schema.sql` and the related
doc claims, or — if Supabase auth is genuinely on the roadmap — relabel
those docs as "planned, not implemented" so they stop describing the
current app incorrectly. This is a documentation decision, not a code
change, and is called out again in the Documentation Report.

## 4. Duplicate logic (extraction candidates for Phase 0/1)

- **Provider connectivity tests** (~2299–2303): `testGroq`,
  `testCerebras`, `testGemini`, `testOpenRouter`, `testOllama` are five
  near-identical functions (`fetch` + try/catch + `.ok` check). Collapse
  into one `testProvider(url, headers)` helper —
  `api/providers/test-provider.js` in the new architecture.
- **Config getters/setters** (~1012–1029): one `get`/`set` pair per
  provider key, all doing the same `localStorage.getItem`/`setItem`.
  Collapse into one `cfgItem(key)` factory — `state/config.js`.
- **Streaming-render pattern**: pipeline (~1602–1626), debate
  (~2443–2448), benchmark (~2555–2560) each repeat "append chunk → render
  markdown with a trailing cursor → scroll into view." One
  `renderStreamingChunk()` helper covers all three —
  `api/streaming.js` / `ui/render.js`.
- **Execution-screen boilerplate**: pipeline/debate/benchmark each clear
  a screen and build result HTML in a loop with near-identical structure.
  Extract a shared `ui/components/execution-panel.js`.

None of these are bugs today — they're working, just repeated. Low risk,
straightforward mechanical extraction.

## 6. Performance

- **No virtualization for the agent list.** `renderList()` (~1233–1255)
  maps every matching agent to a card and joins the HTML in one pass.
  With 247 agents on disk (221 currently wired into the UI — see §7),
  an unfiltered view renders the full set at once. Not currently broken,
  but worth a virtualized/windowed render as the roster keeps growing.
- **O(n²)-ish markdown re-parsing during streaming.** Each streamed
  chunk re-runs `md()` over the *entire* accumulated output so far
  (pipeline ~1605/1612, debate ~2446, benchmark ~2557), rather than
  parsing incrementally. Noticeable mainly on very long responses;
  recommend either incremental parsing or only re-rendering markdown on
  stream completion with a plain-text cursor in between.
- Inline CSS/JS (no bundling/minification) is an intentional and
  reasonable tradeoff for a build-free single file; not flagged as an
  issue, just noted as something Phase 0's CSS extraction should preserve
  (i.e., don't add a minifier — that would reintroduce a build step).

## 8. Accessibility

Minimally accessible, not WCAG-compliant: no `aria-*` attributes or
`role` attributes anywhere, no `<label for=...>` on Creator form inputs
(placeholder text only), no focus trapping in modals, tab order relies on
DOM order (acceptable but not robust). Color contrast is actually good —
the dark theme's CSS variables (`--bg:#0f1117`, `--text:#e2e8f0`,
`--accent:#f59e0b`) give strong contrast ratios by inspection. No images
without alt text (the app uses emoji as text characters, not `<img>`).
Recommend addressing ARIA roles/labels as part of Phase 1 (Workspace
tools), since that's where the highest-traffic interactive elements
(agent cards, chat) live.
